# 大规模报告（月报/年报）处理方案设计

> 状态：设计中（暂不实施） | 日期：2026-07-05

## 一、问题背景

当前架构中，`scanGit` 一次性返回全部原始 commit 数据，全部进入 LLM 对话上下文。对于日报/周报（几十条 commit），这不构成问题。但对于：

- **月报**：可能 200-500 条 commit
- **年报**：可能 1000+ 条 commit

原始 JSON 数据量会直接超出模型上下文窗口（GPT-4.1-mini 约 128K tokens），导致生成失败或结果质量严重下降。

### 当前数据流瓶颈

```
用户说"生成本周周报"
  → scanGit 扫描全部仓库 → 返回全部原始 commit JSON（无截断）
  → 全部塞进 LLM 对话历史
  → MAX_TOOL_ROUNDS = 10（大范围场景不够用）
  → generateReport → LLM 一次性生成报告
```

## 二、交互模式

**混合策略**：小范围自动处理，大范围分步交互。

| 条件 | 模式 | 行为 |
|------|------|------|
| 总 commit ≤ 100 | 一键自动 | 走现有流程，用户等结果 |
| 总 commit > 100 | 分步交互 | 逐仓库/逐时间片处理，中间展示进度 |

## 三、阈值与拆分策略

### 触发阈值

- **判断依据**：总 commit 数量（跨所有仓库汇总）
- **临界值**：100 条
- **判断时机**：扫描完成后，生成前

### 拆分策略（两级拆分）

1. **一级：按仓库拆分**——先处理 commit 少的仓库（快速消化），主力仓库留到最后
2. **二级：按时间拆分**——单个仓库内 commit 仍超 100 时，按周/月切分时间片

```
年报流程示例（总 615 条 commit）：
  projC (15 条) → 直接生成仓库摘要
  projB (80 条) → 直接生成仓库摘要
  projA (520 条) → 超阈值，按 4 周为一片
    → 第 1-4 周摘要
    → 第 5-8 周摘要
    → ...
    → 第 49-52 周摘要
    → projA 汇总（13 片摘要 → 1 个仓库摘要）
  → 全局汇总（3 个仓库摘要 → 最终报告）
```

### 进度可见

每处理完一个仓库或时间片，Agent 输出简短摘要，让用户感知进度：

```
✅ projC（15 条提交）：完成 API 文档更新、修复 2 个偶发 bug
⏳ 处理 projA（520 条提交，分 13 片）...
   ✅ 第 1-4 周：完成用户模块重构、新增支付回调
   ✅ 第 5-8 周：实现消息推送、优化数据库查询
   ...
```

## 四、推荐方案：scanGit 分页 + 分步汇总

### 核心思路

给 `scanGit` 增加三个参数，让 Agent 能按需分页拉取数据，而非一次性全量。

### scanGit 新增参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `countOnly` | `boolean`（可选） | `true` 时只返回 commit 数量，不返回具体数据 |
| `offset` | `number`（可选） | 分页起始位置，0-based |
| `limit` | `number`（可选） | 单页最大条数，建议默认 50 |

### 交互流程

```
1. Agent 先调用 scanGit(countOnly=true) 探测总量
2. 总量 ≤ 100 → 走现有流程（一次 scanGit 全量 + generateReport）
3. 总量 > 100 → 走分步流程：
   a. 按仓库排序（commit 少的优先）
   b. 逐仓库 scanGit(offset, limit)
   c. 每批数据生成一段摘要（非最终报告，是结构化中间结果）
   d. 展示进度给用户
   e. 所有仓库处理完后，汇总摘要生成最终报告
```

### 改动范围

| 文件 | 改动 |
|------|------|
| `src/agent/tools/scanGit.ts` | 增加 `countOnly`、`offset`、`limit` 参数，修改 git log 命令拼接逻辑 |
| `src/agent/prompts/system.ts` | `COLLECT_SYSTEM_PROMPT` 增加大数据量分步处理策略说明 |
| `src/tui/useSession.ts` | `MAX_TOOL_ROUNDS` 调高（年报场景约需 30-50 轮） |
| `src/agent/types.ts` | 新增中间数据结构（仓库摘要、时间片摘要） |

### 不改动的部分

- TUI 层交互逻辑不变（仍是对话式，只是轮次更多）
- `generateReport` 工具接口不变
- 阶段切换机制不变（collect → generate）
- 配置层不变

## 五、与 LangGraph 的关系

引入 LangGraph 后，分步处理可以更优雅地实现：

- **StateGraph 节点化**：每个「探测总量 → 按仓库拆分 → 按时间分片 → 生成摘要 → 汇总」步骤各自是一个节点
- **条件路由**：基于 commit 数量自动选择「直接生成」或「分步处理」分支
- **中断恢复**：分步处理中的用户交互天然对应 `interrupt()` 机制
- **并行处理**：不同仓库/时间片的摘要生成可并行执行

当前的 `scanGit` 分页方案为 LangGraph 迁移预留了基础——分页参数和 `countOnly` 可直接复用为 StateGraph 节点的输入输出。

## 六、决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 交互模式 | 混合（小自动、大分步） | 日报/周报不增加交互负担，大范围有可控感 |
| 触发阈值 | 按 commit 数量（>100） | 直观可控，与时间跨度解耦 |
| 拆分策略 | 先仓库、后时间 | 仓库是天然边界，仓库内时间分片兜底 |
| 进度展示 | 逐片输出摘要 | 用户感知进度，不依赖进度条 UI |
| 实现方案 | scanGit 分页 | 改动集中，不破坏分层，为 LangGraph 预留接口 |

## 七、状态

**暂不实施**。待 LangGraph 引入后重新评估，届时分步处理可作为 StateGraph 的一个条件分支实现。当前文档作为设计方向的备忘录保存。
